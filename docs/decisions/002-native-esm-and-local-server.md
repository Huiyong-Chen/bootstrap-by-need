# 001. 初始技术栈与工程形态

## 背景
项目启动阶段。
业务需求单一：随机展示预设题库中的题目。
目标：以最快速度验证交互逻辑，无需考虑扩展性。

## 决策
1. **采用原生技术栈 (Vanilla Stack)**
   - HTML5, CSS3, ES6+ (运行在现代浏览器中)。
   - 不引入任何构建工具 (No Build Tools)。
   - 不引入任何包管理器 (No NPM/Yarn)。

2. **数据源硬编码**
   - 题目数据作为 Array 存储在 Javascript 内存中。
   - 不进行网络请求，不分离数据文件。

3. **极简目录结构**
   - 采用 `js/` 和 `styles/` 分离代码，保持 `index.html` 干净。
   - 依赖浏览器的原生模块解析或直接脚本加载。

## 决策依据
*   **开发效率**：省去了配置环境的时间，文件创建即开发。
*   **调试体验**：无 SourceMap 干扰，所见即所得。
*   **部署成本**：该阶段产物可直接在任何静态服务器甚至本地文件系统运行。

## 后果与代价
*   **正向**：极速启动，心智负担为零。
*   **负向**：
    *   **全局污染风险**：目前 JS 运行在全局作用域（除非使用 module type），变量容易冲突。
    *   **复用性差**：HTML 片段无法复用（没有组件化）。
    *   **缓存问题**：后续更新代码时，没有哈希文件名，可能面临浏览器缓存问题。
    *   **类型安全**：无 TypeScript，完全依赖开发者的自觉。

**结论**：在当前题目数量少（<100）且无复杂状态交互时，此方案收益最大。一旦涉及多人协作或复杂状态，需立即重构。
