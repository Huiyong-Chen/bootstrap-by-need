# Inner OS: 问题代码的"故意"保留

## 当前阶段任务

记录当前代码状态，暴露问题，为重构做准备。

---

## 架构师的内心独白

当我看到代码中出现越来越多的"坏味道"时，我的第一个冲动是立即修复它们。

"这个拼写错误太明显了！"
"这个作用域问题会出 bug 的！"
"这个变量重复初始化太蠢了！"

但我停住了。

### 纠结：要不要现在就重构？

如果我现在就修复这些问题：
- 我会感觉自己很专业
- 代码会变得更干净
- 可能避免一些潜在的 bug

但我也会失去一些重要的东西：
- **问题积累的真实轨迹**
- **重构时机的客观判断**
- **向其他开发者展示"何时需要重构"的证据**

### 顿悟：问题代码是有价值的

这些问题代码就像是项目演进的"化石记录"：

1. **它们记录了 TypeScript 引入后的真实状态**
2. **它们证明了'按需搭建'的必要性**
3. **它们为后续重构提供了明确的动机**

如果我现在就修复它们，后人看到的就是一个"完美"的代码库，他们不会知道这些完美是如何从混乱中来的。

### 决定：保留，但记录

我决定：
- 故意保留这些问题代码
- 但在提交信息中明确记录它们的存在
- 在文档中详细说明这些问题
- 为后续的重构阶段做准备

### 对未来的期待

当这些问题开始严重影响开发效率时（比如频繁的 bug 报告，或者新功能开发受阻），重构就会成为自然而然的选择。

到那时，我不仅能修复问题，还能向团队展示：
- 为什么这个重构是必要的
- 重构前后的对比
- 问题是如何产生的

这比任何"最佳实践"都有说服力。
