# 按需搭建项目（bootstrap-by-need）

一名架构师搭建项目的内心 OS  
An architect builds the inner OS of a project

---

我并不是一开始就想要 Vite、TypeScript、JSX，  
甚至也不确定这个项目最终会长成什么样子。

我只知道一件事：  
**任何被预设的复杂度，都会在未来反噬我。**

---

## 这个仓库是什么

这是一个**真实存在、可以运行的项目**。  
它记录的不是“我最终用了什么技术”，  
而是：

- 我在什么时候开始感到不舒服
- 是什么问题逼迫我做出一个工程决策
- 我为什么在这一刻引入某种能力，而不是更早、也不是更晚

这里的每一个工具、每一份配置、每一段代码，  
**都不是起点，而是结果**。

---

## 我是如何思考“搭建”的

我不把「搭建项目」理解为：

> 选一个脚手架 → 装一堆依赖 → 开始写业务

在我这里，搭建更像是一次持续的自问：

- 手动刷新是否已经开始拖慢思考？
- 模块路径是否已经变得难以维护？
- 我是否已经需要类型、边界和约束？
- 我现在需要的是工具，还是只是多写几行代码？

**只有当问题真实出现时，工具才有引入的资格。**

---

## 你在这里能看到什么

- 从几乎没有工程能力开始
- 到引入开发运行时（dev runtime）
- 再到 TypeScript、模块系统、构建配置逐步成形
- 每一次演进，都有对应的代码、配置、commit 记录和文档解释

如果你只看最终状态，你会觉得它“理所当然”；  
但如果你顺着 commit 往回走，你会看到：

> 原来复杂度，是这样一步步长出来的。

---

## 如何阅读这个仓库

没有“正确的打开方式”，但我建议：

1. **按 commit 时间线阅读**
2. 对照 `docs/` 中的说明：
   - **`docs/innerOS/`**：架构师的主观思考（当时在纠结什么）
   - **`docs/decisions/`**：客观的架构决策记录（最终决定了什么）
   - **`ENGINEERING-MILESTONES.md`**：完整的工程化里程碑总览
3. 不要跳过“看起来很原始”的阶段

因为真正重要的，  
**从来不是最终选型，而是选型发生的时机。**

---

## 这个项目不承诺什么

- 不承诺最佳实践
- 不承诺适用于所有团队
- 不承诺技术永不过时

它只承诺一件事：

> **这里的每一次复杂化，都是被现实逼出来的。**

如果你也曾在项目一开始犹豫过  
「要不要上构建工具」「是不是该直接上 TS」  
那你大概能理解这里发生的一切。
